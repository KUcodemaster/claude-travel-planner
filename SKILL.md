# 여행 계획 웹페이지 생성 (Travel Planner)

사용자가 준 **여행지 + 기간(또는 날짜)**을 바탕으로, 실제 정보를 웹 검색으로 검증하고 **브라우저에서 바로 열리는 단일 HTML 인터랙티브 여행 일정표**를 만드는 스킬입니다.

This skill turns a destination and travel dates into a polished, self-contained interactive itinerary web page (one HTML file, no server, no external assets).

## 언제 사용하나 (When to use)

- "도쿄로 5일 여행 계획 짜줘", "10월에 파리 일주일 일정 웹페이지로" 같은 요청
- 사용자가 여행지·기간을 주고 일정/동선/맛집을 정리해 달라고 할 때

지명이 없거나 기간이 불명확하면 먼저 물어보세요. 추측으로 만들지 마세요.

## 사전에 확인할 것 (Clarify first)

다음이 불명확하면 `AskUserQuestion`으로 2~4개만 골라 물어보세요 (한 번에):

1. **동행/인원** (혼자·커플·가족(아이 동반 여부)·친구) → 동선·난이도·명소 선택이 달라짐
2. **예산 수준** (절약형·표준·프리미엄) → 숙소·투어·식당 등급
3. **관심사** (자연·명소·미식·액티비티·쇼핑·휴양 등, 복수) → 일정 구성 비중
4. **정확한 날짜/요일** → 마켓·휴무일·시즌 이벤트·일몰 시각에 직접 영향
5. **거점 도시 고정 vs 이동형** → 근교 당일치기를 넣을지

이미 답이 주어졌으면 다시 묻지 마세요.

## 제작 절차 (Build process)

### 1. 실제 정보 검증 (검색 필수)

절대 기억에만 의존하지 마세요. `tavily-search`(있으면) 또는 `WebSearch`/`WebFetch`로 아래를 **반드시 교차 확인**하고, 확인 못 한 것은 일정에 "확인 필요"로 표시:

- 해당 **날짜의 요일**(툴로 계산), 그 요일에 **여는지**(마켓·박물관·식당 휴무일)
- **계절/날씨**, **일몰 시각**, **서머타임 적용일**(특히 봄·가을 경계) — 일몰 관련 일정(선셋 워크 등)에서 흔한 오류
- **이동 시간**(도시↔근교, 명소 간) — 낙관적으로 적지 말 것
- **시즌 이벤트** 개최 날짜가 여행 기간과 실제로 겹치는지
- **예약 필수** 항목(투어·전망대·인기 식당)
- **최근 변경**(시설 이전·폐업·운영시간 변경)

정직 원칙: 겹치지 않는 이벤트, 시기가 이른 개화 등은 "겹치지 않음/이를 수 있음"으로 솔직히 적으세요.

### 2. 데이터 구조 설계

일정을 JS 데이터로 구성합니다. Day별(J, 계획형) + 지역별(P, 탐색형) 두 관점 모두 만드세요.

```js
// 각 Day
{ n:1, date:"...", dow:"요일", theme:"...", illus:"테마키",
  blocks:[
    { time:"09:00–10:30", dur:"1.5시간", t:"장소/활동", note:"설명", map:"구글맵검색어",
      move:1,                       // 이동 구간이면 (선택)
      book:[["플랫폼","URL"], ...], // 예매 링크 (선택)
      eat:{near:"지역", q:"검색어", spots:[["가게","한줄설명","구글맵검색어"], ...]} // 맛집 (선택)
    }, ...],
  tips:[["라벨","내용", 1?], ...] }  // 세번째 값 1이면 강조색
// 지역(P뷰)
{ name:"지역명", illus:"테마키", tag:"짧은 태그", desc:"설명",
  places:[["장소","한줄","구글맵검색어"], ...],
  eats:[["가게","한줄","구글맵검색어"], ...] }
```

**시간 정보 필수**: 각 블록에 실제 시각(`time`)과 소요(`dur`)를 넣고, 장소 간 이동은 별도 `move` 블록으로 분리해 이동수단·소요를 명시.

### 3. HTML 생성 (`assets/template.html` 참고)

이 스킬 폴더의 `assets/template.html`이 완성형 참고 구현입니다(시드니 예시 데이터 포함). 이를 뼈대로:

- `DAYS`, `AREAS` 데이터만 새 여행지로 교체
- 팔레트·일러스트는 **여행지의 정체성**에 맞게 조정(아래 디자인 원칙)
- 그대로 복사하지 말고, 목적지에 맞는 색·일러스트·문구로 다시 만드세요

포함 기능(구현은 template.html에 있음):
- **J(계획형)/P(탐색형) 모드 전환** — J는 Day별 시간표, P는 지역별 나열
- **Day 탭** + 이전/다음, **체크리스트 진행률**(localStorage 저장)
- **테마별 SVG 일러스트**(외부 이미지 없이 인라인 벡터 — CSP·저작권 안전)
- **구글맵 링크**(장소·맛집), **예매 버튼**(공식+플랫폼 하이브리드, 폴백으로 검색 링크)
- **진입 인트로 애니메이션**(선택), **반응형**(모바일 뷰), 다크/라이트 테마
- **PWA(설치·오프라인)** — 아래 3-1 참고

### 3-1. PWA (프로그레시브 웹 앱) — 단일 HTML 인라인 방식

여행 페이지는 **여행지에서 오프라인으로 볼 일이 많으므로 PWA를 기본 포함**합니다. 단일 HTML을 유지한 채 파일 분리 없이 인라인으로 구현합니다(구현은 template.html의 마지막 `<script>` 블록에 있음):

- **manifest**: JS에서 객체를 만들어 `data:application/manifest+json,...` URI로 `<head>`에 주입. 아이콘은 외부 이미지 대신 **이모지를 SVG data URI**로(테마색 배경 + 이모지). `name`/`theme_color`/`background_color`/`icon 이모지`만 여행지에 맞게 바꾸면 됨
- **Service Worker**: SW 코드를 문자열로 만들어 **Blob URL로 등록**. network-first + 캐시 폴백으로 방문 페이지를 오프라인 제공. `location.protocol`이 http(s)일 때만 등록(파일 열기 `file://`에선 스킵)
- **설치 버튼**: `beforeinstallprompt`를 잡아 우하단 "앱으로 설치" 버튼 노출, iOS 등 미지원 시 "공유 → 홈 화면에 추가" 토스트 안내. standalone으로 이미 열렸으면 버튼 숨김
- iOS 대응: `apple-touch-icon`, `apple-mobile-web-app-capable` 메타 추가

주의: **오프라인·설치는 https 배포(또는 localhost)에서 동작**하고 `file://`·Artifact 미리보기에선 제한됩니다. 배포 대상이 정적 호스팅(Vercel 등)일 때 실효가 있습니다.

### 4. 디자인 원칙 (frontend-design 준수)

- AI 기본값(크림+세리프+테라코타, 보라-파랑 그라디언트) 회피. **여행지 고유의 색**을 팔레트로 인코딩
- 뉴트럴은 강조색 쪽으로 살짝 편향, 다크/라이트 둘 다 설계
- SVG 일러스트는 `viewBox`+`aspect-ratio`로 반응형(고정 height 금지, `slice` 대신 `meet`)
- 폰트 CDN은 CSP에 막히니 시스템 폰트 스택 사용, 개성은 타입스케일·색·레터스페이싱으로

### 5. 검증 (배포 전 필수)

- `<script>` 블록을 추출해 `node --check`로 JS 문법 확인
- 데이터 정합성: 모든 블록에 시각 존재, Day별 블록 수, 예매/맛집 링크가 https인지
- 링크 URL은 검색에서 실제로 나온 것만 사용(지어내지 말 것). 봇 차단(403)은 죽은 링크가 아님

### 6. 산출/공유

- **로컬 확인**: 파일을 열어보도록 경로 안내
- **Artifact 게시**(claude.ai): `<head>` 자동 래핑되므로 `<title>+<style>+body`만
- **Vercel 등 정적 배포**: 완전한 HTML 문서(`<!doctype>`, `<head>`에 **viewport 메타 필수** — 없으면 모바일이 데스크톱 폭으로 렌더)로 감싸서 `index.html`

### 7. (권장) 타당성 자체 검토

완성 후 **서브에이전트(general-purpose)**에게 일정 타당성 검토를 맡기면 품질이 크게 오릅니다: 요일 정합성·이동시간 현실성·시간 배분·계절/일몰·놓친 예약·동선 효율을 웹 검색으로 교차 확인시키고, 지적을 반영해 재생성하세요.

## 오류 처리 (Errors)

- 검색 도구가 없거나 실패 → 그 사실을 알리고, 확인 못 한 정보는 일정에 "현지에서 확인" 표기. 없는 사실을 지어내지 마세요.
- 외부 이미지가 필요해 보여도 Artifact는 CSP로 막습니다 → 반드시 인라인 SVG로.
- WebFetch가 특정 사이트(예: naver 계열)를 못 가져오면 검색 스니펫으로 대체.
